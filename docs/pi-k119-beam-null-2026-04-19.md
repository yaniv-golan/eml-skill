# `pi` witness — beam-search null result at K<=119 (2026-04-19)

## Summary

A 90-minute targeted beam-search for the constant `pi` under `max_k=119`,
`per_level_cap=500000`, `seed_witnesses=True`, `seed_subtrees=True`, and
`domain=complex-box` returned **no match**. The shipped K=121 upper bound
(`pi = mult(sqrt(neg(1)), NIPI)`, cascaded from the K=75 i/sqrt-neg-1
composition — see `docs/pi-alternate-identities-2026-04-19.md` and
`docs/i-k75-cascade-2026-04-19.md`) therefore stands. The paper's
direct-search floor of >53 is not refuted, but neither is it closed.

A companion 30-minute probe at `max_k=79` with the same cap/seeding
returned null as well; that frontier (K=79, candidates evaluated =
2,600,563) is recorded in the per-K table below for completeness.

## Command

```
PYTHONPATH=eml-skill/skills/_shared \
  python3 eml-skill/skills/eml-optimize/scripts/optimize.py search \
    --target pi --max-k 119 --time-budget 5400 --per-level-cap 500000 \
    --domain complex-box --seed-witnesses --seed-subtrees --format json
```

Symbolic gating was intentionally skipped: `pi` is not registered in
`eml_core.symbolic.SYMBOLIC_TARGETS` (which covers the closed-form
elementary targets — `neg`, `inv`, `exp`, `ln`, `e`, `add`, `sub`, `mult`,
`pow`), so the `--symbolic-gate` flag would have no effect for this target.

## Outcome

- **Found:** `false`
- **Stopped reason:** `time-budget` (5471.6 s wall-clock against a 5400 s
  budget — the process returned on the next level boundary after the
  timer tripped).
- **Candidates evaluated:** 3,100,701 unique function vectors.
- **Best K:** `n/a`.

## Interpretation

1. **The cap chokes K=17..27 before the enumeration can widen.** At
   `per_level_cap=500000` every level from K=17 through K=27 saturates —
   the raw bottom-up pool of unique function vectors grows faster than
   the admission cap lets through. The larger 90-minute budget got us
   one additional saturated level (K=27) relative to the 30-minute
   stretch (where the budget ran out at K=25 saturation), but no
   further. Each additional saturated level roughly doubles the
   combinatorial pool downstream without doubling the fraction of the
   pool that survives the goal-set priority — so the budget would have
   to grow by orders of magnitude to push the bottom-up frontier past
   K=27.

2. **High-K entries are seeded subtrees, not newly-composed candidates.**
   The per-K counts at K=29 and above (single digits, max 10 at K=99)
   are pre-populated library subtrees from `sqrt`, `neg`, and the K=121
   pi tree's own internal decomposition, retained via
   `--seed-subtrees`. Genuine bottom-up enumeration never reached those
   levels within the budget. The meet-in-the-middle `_generalized_
   targeted_scan` and backward goal propagation probed each of those
   seeded candidates against the K=17..27 pool and found no complement
   that reconstructs the pi function vector.

3. **`pi = mult(sqrt(neg(1)), NIPI)` has no obvious shorter witness path
   through the current library.** The shipped K=121 tree is already
   1+16 tokens below the published K=137 nine-stage proof-engine
   construction (the `i = sqrt(neg(1))` cascade at K=75 replacing the
   K=91 i). Every hand-checked alternate identity (see `docs/pi-
   alternate-identities-2026-04-19.md`: `mult(neg(i), ln(-1))`,
   `2·asin(1)`, `4·atan(1)`, `acos(-1)`, `div(ln(-1), i)`, etc.) lands
   at K>=145. The beam confirms that no K<121 structural shortcut is
   reachable through sqrt/neg/exp/ln/add/mult/`NIPI`-subtree building
   blocks within the 90-minute budget at this cap.

4. **Paper K>53 floor is consistent.** Our empirical floor at
   `per_level_cap=500000` is >=119 (i.e. the search cannot exhibit a
   K<=119 witness for `pi` under these parameters). The paper's
   reported direct-search lower bound of >53 therefore still sits
   comfortably below our upper bound of 121; the gap 54..121 is
   unresolved.

## No cascade

Because no K<121 witness was discovered, the existing `pi` tree is
unchanged, and no dependents are affected (`pi` is not a subtree of any
other witness in the library). No mutation to `witnesses.py`,
`tests/test_witnesses.py`, or `docs/leaderboard.md` is required.

## Next steps (not taken in this session)

- Raise `per_level_cap` to 1,000,000 or 2,000,000 and grant a multi-hour
  budget, specifically targeting the K=17..29 bottleneck where the cap
  currently clips. Memory use at cap=500k already hit ~5.5 GB; a 2M cap
  would need a larger-memory machine or a disk-backed deduplication
  scheme.
- Extend `SYMBOLIC_TARGETS` to cover `pi` (sympy `pi`) so the symbolic
  gate can promote near-misses at intermediate K into verified matches
  when hash collisions hide true hits.
- Attempt a hand-compiled short-pi via closed-form identities that the
  sympy-based compiler does not currently try — though
  `docs/pi-alternate-identities-2026-04-19.md` already enumerated the
  obvious library-composition candidates, all of which land above K=137.
  A genuine new primitive `i`-free identity (or a bespoke pi-specific
  proof harvested from a new proof-engine stage) would be the next
  frontier.
- Revisit once any upstream witness (sqrt, neg, NIPI-subtree, or the
  underlying ln/mult/add trees) shortens — every token saved on an
  embedded subtree propagates linearly into pi's K.
